# Vibe E2E — 產生並執行 vibe spec（v2）

## 目的

把 vibe 階段新增的**互動行為**與**結構新增**自動轉成 Playwright spec，並執行驗證。

不測：純 visual、業務合約（那是 `/vibe-check` 主 spec 的工作）。

**產出的 spec 是 keep 的**：既然判斷值得生成 e2e，就值得長期守。落地後會進 gate 守門（`/vibe-check` 與 pre-push 都跑 `playwright.gate.config.ts`，涵蓋 `test/e2e/vibe/`；執行環境差異見 vibe-check「目的」段）。時序敏感、無法穩定綠的 spec 產到 `test/e2e/vibe/unstable/`，不進守門、手動跑 /vibe-e2e 或 vibe config 時照跑。

## 前置條件（軟性 gate）

執行前**應確認**：

1. **`/vibe-check` 已綠燈**——若主 spec 紅燈，vibe 改動本身可能違規，vibe spec 結果意義不大
2. 已用 `/vibe-setup` 看過分層報告（可選但建議）——確保理解這次會產哪些 spec

執行時若**無法確認**這次 session 已跑過 vibe-check，請先告知使用者並建議補跑，使用者點頭再繼續。

## 使用方式

```bash
/vibe-e2e            # 對比 HEAD：產 spec + 跑
/vibe-e2e <ref>      # 指定 base ref
/vibe-e2e --dry-run  # 只產 spec、不跑（讓使用者預覽）
```

---

## 絕對禁止

- 不可動 `test/e2e/specs/`（主 spec 凍結）
- 不可動 `spec/gherkin-feature/`、`spec/e2e-flows/`
- 不可動 `playwright.config.ts`（主 config 凍結；vibe 用 `playwright.vibe.config.ts`）
- 不可主動 commit / push
- 不可在 `test/e2e/vibe/` 之外寫入

## 流程

### Step 0：前置確認

確認這次 session 已跑過 `/vibe-check` 綠燈。若無紀錄，提示使用者「建議先跑 /vibe-check 確認主 spec 綠燈」，等使用者點頭。

### Step 1：分類 vibe diff

```bash
git diff <base> -- app/pages/ app/components/ app/layouts/
```

分類規則（grep pattern、sub-pattern mapping、邊界 case）見 [`../vibe-setup/classification.md`](../vibe-setup/classification.md)，與 `vibe-setup` 共用。

**只處理「互動」與「結構」hunk**，純 visual 略過。

### Step 1.5：載入合約 testid 白名單

```bash
# 提取主 spec 所有 getByTestId 字串作為合約白名單
SPEC_TESTIDS=$(grep -rhoE "getByTestId\(['\"]([^'\"]+)['\"]\)" test/e2e/specs/ \
  | sed -E "s/.*['\"]([^'\"]+)['\"].*/\1/" | sort -u)
```

這個集合 = vibe spec 可直接使用的 **testid 合約白名單**。之後 Step 3 套 pattern 模板時，locator 推測規則會用這個集合做檢查。

### Step 2：盤點既有 vibe spec（keep，不清空）

vibe spec 預設 **keep**，不做全清重生。只處理「同來源」的覆蓋：

1. 讀既有 `test/e2e/vibe/**/*.spec.ts` 的首行 marker（`Source hunk: {path}:{lines}`）
2. 本次 diff 的 hunk 若與既有 spec **同來源檔案 + 同 pattern** → 重生**覆蓋**該檔（視為「透過重生編輯」）
3. 其他既有 spec 一律保留——它們是 gate 守門的一部分，刪除是使用者的決定，不是 skill 的

**絕不可因為既有 vibe spec 紅燈就刪掉它**。紅燈處理選項（修 UI / 更新 / 刪除 / 搬 unstable/）要列給使用者選。

### Step 3：依 pattern 套版產生 spec

對每個互動 / 結構 hunk，依 sub-pattern 套對應模板（見 [`references/patterns.md`](references/patterns.md)），產出 `test/e2e/vibe/{layer}-{module}-{pattern}-{n}.spec.ts`。`{layer}` 取分類層：互動 → `interaction`、結構 → `structure`（`interaction` 只作 layer 前綴用，不是 sub-pattern 名）。

每個檔案**首行加 marker**：

```ts
// AUTO-GENERATED by /vibe-e2e — edit via regenerate（/vibe-e2e 同來源覆蓋）
// Source hunk: app/pages/practice/live.vue:120-135
// Pattern: toggle
```

不確定的 locator 用 `// TODO:` + `.skip` 標記（不阻塞綠燈判定）。

**生成品質規則（spec 要進守門，必須耐 flaky）**：

- **禁用 `page.waitForTimeout`**——一律事件等待：`expect(...).toBeVisible()` 自帶重試、`page.waitForFunction`（等 DOM 計數/狀態變化）、`waitForResponse`（等 API）
- 斷言寬緊有度：總數會浮動的（合併資料、分頁）用**下限斷言**（`toBeGreaterThanOrEqual`），不要寫死精確值
- 可容忍暫時停滯的輪詢用 stall 計數器，不要一次沒變化就放棄
- **時序敏感的 pattern**（SSE / 即時推播 / 依賴真實時間差的行為）→ 產到 `test/e2e/vibe/unstable/`，並在回報中說明「此 spec 不進守門」

### Step 4：跑 vibe spec + 穩定性驗證

```bash
# 全量 vibe spec（含 unstable/，手動跑時照跑）
npx playwright test --config playwright.vibe.config.ts
```

**本次新生成/覆蓋的 spec 必須連跑驗證穩定**才算完成（一次綠可能只是運氣）：

```bash
npx playwright test --config playwright.vibe.config.ts {新檔名} --repeat-each=3
```

連跑不穩的不准落地進守門區：修等待邏輯，修不動就搬 `unstable/` 並在回報中說明。

`--dry-run` 模式跳過此步。

### Step 5：回報

**綠燈**：

```
=== Vibe E2E 通過 ===

產生 5 個 vibe spec（覆蓋 1 個同來源舊檔、保留 6 個既有檔、0 個產到 unstable/）
跑了：8 cases，6 passed ✅，2 skipped ⚠️（明細見下方「被 .skip 清單」）
新生成 spec --repeat-each=3 連跑：全綠 ✅（已可進守門）

涵蓋：
- interaction-practice-toggle-1：sidebar 折疊 → 通過
- interaction-players-tab-1：球員 tab 切換 → 通過
- structure-players-empty-state-1：空狀態顯示 → 通過
- structure-accounts-loading-state-1：loading UI → 通過
- structure-cameras-error-state-1：error UI → 通過

vibe 改動的互動/結構驗證通過，可以 commit。
```

**紅燈**：

```
=== Vibe E2E 失敗 ===

8 cases，2 failed ❌

失敗清單：

1. interaction-practice-toggle-1.spec.ts › vibe：sidebar 折疊
   失敗訊息：點觸發按鈕後 sidebar 仍 visible
   來源 hunk：app/pages/practice/live.vue:120-135
   pattern：toggle
   嫌疑：toggle 邏輯沒接好，或目標元件 locator 推測錯誤
   建議行動：
   - 對照 app/pages/practice/live.vue 確認 ref 與 @click 綁定
   - 若 locator 推測錯誤，請手動跑 npx playwright test --config playwright.vibe.config.ts --ui 用 codegen 找正確 locator

2. ...

下一步建議：
- 修 UI 後重跑 /vibe-e2e 驗證
- 若 pattern 推測明顯不適用，請手動跑 /vibe-check 確認主 spec 仍綠，並考慮 vibe 改動是否要保留
```

**被 .skip 清單**（無論綠紅都要列於 Step 5 輸出末尾；**每個 .skip 都要有一列，不可只報數字**）：

.skip 不是通過——它代表**這塊 UI 沒被測到**。四種原因類別都要能對應：

- `orphan-testid`：用了不在主 spec 合約白名單的 testid
- `no-locator`：新增元素無語意 role/name、也無 testid
- `unknown-trigger`：條件渲染推不出「怎麼觸發」
- `hover-visual`：hover 類效果，自動測意義低（通常維持 skip）

```
⚠️ 被 .skip 清單（2 個）：

[orphan-testid] app/components/Sidebar.vue:30 → 'sidebar-collapse-button'
  來源 hunk：app/components/Sidebar.vue:28-34　pattern：toggle
  解除：
    (a) 改用 getByRole('button', { name: /折疊/ }) — 推薦
    (b) 若這是長期合約，改 .flow.md → /test e2e spec → 主 spec 收進後再啟用

[unknown-trigger] app/pages/practice/index.vue:120 → 條件渲染推不出觸發步驟
  來源 hunk：app/pages/practice/index.vue:118-130　pattern：conditional-render
  解除：補上觸發條件的操作步驟（例：篩選某類別 / 改某輸入），再重跑 /vibe-e2e

合約 testid 是業務合約的可定位表面，不應由 vibe 階段自由擴張。
逐條決定「補定位 / 補觸發」或「維持 skip」，不要靜默略過。
```

## Pattern 模板表

10 個 sub-pattern（toggle / tab / modal-toggle / conditional-render / hover / state-change / loading-state / error-state / empty-state / new-region）各對應一份模板，見 [`references/patterns.md`](references/patterns.md)——含「固定值皆為示意、落地前依專案替換」的鐵律，套版前必讀。分類標籤與模板一一對應：classification.md 新增 sub-pattern 時，這裡的數字、清單與 patterns.md 模板要同步。

---

## Locator 推測規則

從 diff hunk 提取：

| 來源 | 推測規則 |
|---|---|
| `<UButton>觸發文字</UButton>` 新增 | `getByRole('button', { name: /觸發文字/ })` |
| `aria-label="X"` 新增 | `getByRole('?', { name: /X/ })` |
| `<UTabs :items="[{ label: 'X' }, ...]">` | `getByRole('tab', { name: /X/ })` |
| `v-model="open"` + `<UModal>` | `getByRole('dialog')` |
| 新 `<section aria-labelledby="X">` | `getByRole('region', { name: /X/ })` |
| 沒有語意 role/name 線索，UI 既有 testid **在白名單內** | `getByTestId('xxx')`（合法，合約一部分） |
| 沒有語意 role/name 線索，UI 既有 testid **不在白名單** | **不可使用**——產 `.skip` + WARNING：「孤兒 testid 候選，請先決定要納入合約還是改用語意 anchor」 |
| 沒有語意 role/name 線索，UI 也沒任何 testid | **不可自己創 `vibe-xxx`**——產 `.skip` + TODO：「此 element 缺合約定位手段，請補 .flow.md 或加語意 anchor」 |

> **與舊版的差別**：舊版 fallback 是「自己創 `vibe-{描述}` testid 並要 UI 補上」——這個路徑會把任意 vibe 階段的 testid 漂白進合約。新版禁止此路徑，未授權 testid 必須先走 flow.md → spec 重生流程。

## 生命週期（keep）

- vibe spec 落地後**長期保留**並進 gate 守門，不再每次全清重生
- 「編輯」一律透過重生：同來源 hunk + 同 pattern 的 spec 由 /vibe-e2e 覆蓋
- **孤兒回收**：每次執行盤點既有 vibe spec 時，依首行 `Source hunk: {path}:{lines}` marker 檢查——(a) 來源檔已不存在，或 (b) 來源檔已重構、該行段找不到對應 pattern 痕跡 → 列入「孤兒候選」清單（附當前紅綠狀態）給使用者，選項：刪除／搬 `unstable/`／保留
- 刪除 / 搬 `unstable/` 是使用者的決定——skill 只建議、列選項，不擅自執行
- 業務合約等級的行為仍應寫到主 spec（走 .flow.md → /test e2e spec 流程），不要塞 vibe 資料夾

## 與相關 skill 的關係

```
/vibe-check    主 spec 守門（先跑、確認綠燈）
   ↓ green
/vibe-setup    分類 + pattern 標記（人類看，純 console）
   ↓ 決策
/vibe-e2e      （這個 skill）依當下 diff 重新分類 + 套版生 spec + 跑
```

`/vibe-e2e` 內部會重跑分類（不依賴 `/vibe-setup` 落地檔，因為前者選了「不落地」設計）。`/vibe-setup` 的角色是給人類預覽決策，不是 `/vibe-e2e` 的前置輸入。

---

## 實作要點

1. **vibe spec 是 keep 的 auto-gen**：同來源重生覆蓋、其他保留，落地即進 gate 守門
2. **pattern 推測寧可保守**：不確定就 `.skip + TODO`，不要硬生錯誤斷言
3. **locator 推測有層次**：role+name 最佳、testid fallback、不確定就 TODO
4. **失敗回報要可行動**：列出來源 hunk、pattern、嫌疑、建議行動
5. **不污染主 spec**：絕不寫入 `test/e2e/specs/`，不修主 config
