# 需求拆解指南（user story + 時間線 + 工種歸屬）

拆解前先把「使用者實際怎麼操作」跟人對齊。**沒對齊 user flow 就標工種，載體與時序會猜錯、歸屬會標反。**

---

## 為什麼先對齊再拆

直接從需求文字跳到「分工種／分畫面」會漏掉三類事：

- **載體猜錯**：把「即時回放」當成場邊 KIOSK，其實是 LINE LIFF → 呈現形態與歸屬全換掉。
- **時序混淆**：把「即時」和「賽後」混成一條 flow → 漏看登入、漏看推播。
- **角色介入點漏看**：會員／登入到底在哪一步介入沒講清 → user story 缺一大塊。

這些都是**量級反轉變數**，只能靠跟人對齊 user flow 鎖定，不能事後腦補。

---

## 步驟 1：萃取四要素

| 要素 | 問什麼 | 常見選項 |
|------|--------|----------|
| **Actors（角色）** | 誰會用、各自要什麼 | C 端使用者（細分身分）、B 端管理者、系統／演算法 |
| **載體** | 在哪個裝置／容器裡操作 | KIOSK 平板、LINE LIFF、手機 web、桌機後台、原生 App |
| **觸發方式** | 這個動作怎麼被啟動 | 人手動點按、**演算法自動偵測**、排程、收到連結／推播 |
| **時間線** | 發生在什麼時間脈絡 | 即時／現場（同步）、賽後／事後（非同步）、隨時（後台） |

> 載體 × 觸發 × 時間線三者一旦講錯，工種就標錯。優先把這三項問清楚。

---

## 步驟 2：拆時間線（最容易出錯的一步）

**同步與非同步要拆成不同時間線，不要混成一條。**

| 時間線型態 | 特徵 | 常見流程重點 |
|------------|------|--------------|
| 即時／現場（同步） | 低延遲、人在現場、零跳轉 | 即時呈現、推播接收 |
| 賽後／事後（非同步） | 連結、推播、登入後才看 | 落地、登入、列表、分享 |
| 隨時（後台） | 管理者操作、無即時壓力 | CRUD、監控、設定 |

### Worked example：匹克球

```
打球當下 ─► 時間線 A：即時爭議回放（LINE 內，零跳轉、不登入）
              觸發＝演算法偵測爭議 → 推送
比賽結束 ─► 時間線 B：賽後精彩回放（推送 → 看片 → 分享）
              會員／登入若有介入，在這條線，不在 A 線
隨時    ─► 時間線 C：B 端管理者後台
```

一開始沒拆時間線時的錯：把 A 當 KIOSK、把會員當單純 LINE 綁定、A/B 混一條。拆開後全部現形。

## 步驟 3：硬停確認（量級反轉變數清單）

把以下變數攤給使用者逐項確認，**任何一項沒鎖定就停在這裡、不要往下拆**：

| 變數 | 不鎖定的後果 | 提問範本 |
|------|--------------|----------|
| **載體** | 呈現形態整個換掉 | 「這個畫面跑在哪？KIOSK／LINE LIFF／手機 web／後台？」 |
| **觸發方式** | 時間線與觸發源標錯 | 「誰觸發？使用者手按，還是演算法／系統自動？」 |
| **平台呈現方式** | 把「後端推原生內容」當成「自建前端」 | 「LINE 上是原生訊息推播（前端 0），還是開 LIFF 自建畫面？」 |
| **LINE 功能歸屬** | 把 Bot 對話流標成前端 | 「這段是 LIFF 網頁，還是 Bot 選單問答？」 |
| **登入／會員介入時機** | 漏掉登入／帳號那條 flow | 「哪一步開始要登入？看 X 之前要不要先是會員？」 |
| **跨系統勾稽** | 綁定流程漏一半 | 「會員全新做，還是跟既有系統勾稽？OAuth／SSO 還是查詢 API？」 |
| **B 端畫面範圍** | 臆造不存在的畫面 | 「後台有哪幾個畫面？逐一講，沒講到的不放進來」 |

用 `AskUserQuestion` 一次問 2–4 題（挑當前案子真正模糊的）。確認／補正後才往下拆。

---

## 步驟 4：工種歸屬判準

標 user story 工種時，看「誰交付這段」：

| 需求特徵 | 工種 |
|----------|------|
| 畫面、互動、資料呈現、串接 client | **前端** |
| 模型偵測／辨識／訓練（落點、事件、姿態…） | **演算法** |
| API、DB、推播發送、OTP 發送、狀態機、媒體伺服器 | **後端** |

**LINE 特別判準（最容易標反）：**

| LINE 功能 | 工種 |
|-----------|------|
| LIFF 網頁（自建 HTML／JS 畫面） | 前端 |
| Bot 對話流（rich menu 選單、輸入問答、OTP 驗證碼、狀態機綁定） | 後端 |
| 推送發送 | 後端（前端只在 LIFF 內呈現） |

**平台影音呈現（做法光譜，前端佔比 0 ↔ 重）：**

| 呈現方式 | 能力 | 工種 |
|----------|------|------|
| 原生 video message／Flex video | 一般播放 + Done／Replay，**無變速／逐幀／落點互動** | 後端推（前端 0）|
| LIFF 自建播放器 | 變速／逐幀／時間軸／canvas 落點都能做 | 前端 |

判準：慢動作／落點是**燒進成片**（後端合成，前端 0）還是**使用者互動操作**（LIFF，前端）。

> **通則**：拆「在第三方平台（LINE／YouTube／嵌入）上呈現 X」時，先確認用**平台原生能力**（前端 ≈ 0）還是**自建前端**——別預設有前端畫面。

---

## 步驟 5：轉 User Story 卡片 + 工種歸屬

每條 flow 的每一步轉成一張卡，句末標工種：

```
US-{時間線}{序號}（{端 C／B}）作為 {角色}，我想要 {操作}，以便 {目的}。 [工種]（依賴 US-xx）
```

`（依賴 US-xx）`為可選尾註——只在真有相依（沒做 xx 這條就動不了）時標，不強制每張卡。

範例：

- US-A1（C）作為打球的玩家，我想要不做任何操作就在 LINE 收到爭議回放，以便當下判斷。 [後端]
- US-B1（C）作為第一次來的民眾，我想要申請 OA 會員並綁定原會員系統，以便兩邊認得是同一人。 [後端]（Bot 對話流）
- US-B2（C）作為打完球的玩家，我想要賽後自動收到精彩回放推播，以便回味與分享。 [後端]（依賴 US-B1 綁定）
- US-C1（B）作為管理者，我想要在後台建立比賽、監看攝影機狀態，以便掌握現場。 [前端]

---

## 步驟 6：輸出格式範本

```
## Actors
- C 端：{玩家…}
- B 端：{管理者…}

## 時間線 User Flow
- 時間線 A（即時／現場）：{觸發 → 步驟 → 結果}
- 時間線 B（賽後／非同步）：{推送 → 步驟 → …}
- 時間線 C（後台／隨時）：{…}

## User Story
- US-A1（C）作為 X，我想要 Y，以便 Z。 [前端]
- US-A2（C）…  [後端]（依賴 US-A1）
- US-C1（B）…  [前端]

## 範圍外（Out of Scope）
- {使用者明說不做的}
- {需求沒提到、刻意未納入的——不臆造}
```

---

## 複雜度自適應

| 案子型態 | 拆解力度 |
|----------|----------|
| 純 B 端 CRUD 後台、單一載體、無即時 | flow 三行帶過，story 列幾條，快速確認即可 |
| 多端／多時間線／載體不明／含即時或勾稽 | 完整時間線 + 多 actor + 逐項硬停確認 |

判準：**只要出現「即時、推播、LINE、登入、會員勾稽、跨系統、平台原生影音」任一訊號，就走完整版。**
