# 難易度評分 + 人天錨定表

兩件事：**(A)** 用因子判難易度；**(B)** 把難易度 / 介面類型對到人天區間。校準順序永遠是「先 RAG，撈不到才用本表」。

> 單位慣例：**人月 = 人天 ÷ 22**。本表人天皆為**前端**佔比（全端工項已切掉後端）。

---

## A. 難易度因子

逐個介面看下列因子，取最高者定級（任一因子踩到高，整體至少高）。

| 因子 | 低 | 中 | 高 | 極高 |
|------|----|----|----|------|
| 影音 / 視覺化 | 無 | 一般播放 / 圖表 | 變速播放、座標映射、即時串流 | 多路同步、複雜編輯器 |
| 即時性 | 無 | 輪詢 | WebSocket/SSE 推播 | 硬 KPI（喚醒 < Ns）+ 即時 |
| 第三方整合 | 無 | 單一成熟 SDK | LINE LIFF / 金流 / 會員 | 多方串接 + 不確定規格 |
| 狀態 / 流程複雜度 | 單頁 | 跨頁共享 | 多步驟流程 + 角色分權 | 複雜編輯器狀態 |
| UIUX 探索 | 套既有 pattern | 小幅客製 | 全新介面需設計 | 創新互動 / 大量設計 |
| 需求模糊度 | 明確 | 局部模糊 | 大量待確認 | 方向未定 |

**難易度 → 基準前端人天區間（單一介面，校準失敗時的粗 band）**

| 難易度 | 區間（人天） | 典型 |
|--------|--------------|------|
| 低 | 1 – 4 | 純表單 / 列表 / 確認頁 |
| 中 | 5 – 12 | 操作介面 + API 串接、LIFF 頁、帳號管理 |
| 高 | 13 – 24 | 視覺化編輯、報告頁、Web 服務介面 |
| 極高 | 30 – 45+ | 多視角播放器、複雜編輯器、會員系統全包 |

## C. 風險乘數

乘數作用在區間的**悲觀端**（樂觀端維持基準）。

| 風險 | 乘數 | 何時套 |
|------|------|--------|
| RWD（一般） | ×1.25 | 介面需手機 + 桌機 |
| RWD（行動需重新設計） | ×1.5 | 複雜表單 / 表格在小螢幕 |
| 需求高度模糊 | ×1.2 – 1.5 | 待確認項多、方向未定 |
| 即時硬 KPI | ×1.2 – 1.5 | 喚醒 < Ns、低延遲串流 |
| 新領域 / 無前例介面 | ×1.3 | 內建表與 RAG 都無同類 |

## D. RAG 校準操作指引

1. 對每個介面，用 `mcp__internal-research-rag__search_documents` 查：
   - `query`：介面語意關鍵字（如「影像播放器 多視角 即時 前端 人天」「帳號管理 權限 介面」「LIFF 頁面 綁定」）
   - `topic: PROJECT`
2. 命中的真實案人天 = **錨點**；同類取 2–3 筆抓區間，越同領域（體育 / 影像 AI / 場館）權重越高。
3. 對齊規則：
   - RAG 錨點與本表差距大時，**以 RAG 為主**（它是現況），本表只兜底。
   - 找到的若是「含後端」的合計，要按屬性切出前端佔比再用。
4. 在輸出「備註」標來源：`RAG:案名` 或 `內建表`。
