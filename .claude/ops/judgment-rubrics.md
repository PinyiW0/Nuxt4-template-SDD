# 判斷 Rubric：拿不定主意時照這裡判

> 每節格式：判準 → 正例 → 反例。正反例多取自本專案真實事件。
> 這份檔案回答的是「怎麼判斷」；「怎麼派工」見 [model-dispatch.md](model-dispatch.md)。

## 1. 何時升級模型

**判準**：升級的訊號是「錯的方式」不是「錯的次數」。同一種錯重複出現 = 模型能力不足，升級；每次錯得不一樣 = 任務定義不清，先修 prompt 再說（升級也救不了爛 prompt）。

- 正例：sonnet 兩次都把 `useFetch` 的 `data`（shallowRef）當深層響應式用，錯法相同 → 這是框架知識盲點，帶失敗軌跡升 opus
- 反例：sonnet 第一次找錯檔案、第二次改錯位置，錯法不同 → 檢查交辦 prompt 發現沒給驗收條件，補上後 sonnet 一次過。升級 opus 是浪費

## 2. 何時算真的完成（完成定義分型）

**程式碼**：**第 5 節分層表判定「該跑哪幾層」，該跑的層全綠才算完成**（此處不重列各層門檻——重列必漂移）。**該跑的層有一層沒跑就宣告完成 = 未完成**。最常被漏的是層 3（`/sdd-review`）與層 4（read-back）——eslint/typecheck/Playwright 抓不到框架語意、授權死角、規範有沒有消費點，那正是它們存在的理由。

**文件／規範**：能具體回答「誰會讀、何時載入、經什麼機制」。指不出消費點 = 未完成。

**修共用建設**（composable、skill 引用檔、規範檔）：列出全部消費點並逐一驗證後才算完成。

- 正例：新增視覺層級規範時，同步接進 `feature-to-ui` skill 的 `@` 引用與 `rules/ui-conventions.md` 的 paths 觸發，兩個消費點都能載到 → 完成
- 反例：把驗收標準規則寫進一份新 md，但沒有任何 skill、rule、CLAUDE.md 索引引用它 → 寫了等於沒寫，不算完成（本專案曾踩過同型的雷：改共用建設沒同步驗證下游產生器 skill；教訓原文在使用者層記憶，不隨 repo）

## 3. 何時停下來問使用者

**必停清單**（命中任一就停，列選項問）：

1. 需要動凍結區才能達成目標（路徑清單以 `.claude/rules/frozen-paths.md` 為準，此處不重列——hook 會擋，別寫 sentinel 繞過）
2. 需要動 `maintenance.md` 第 1 節「動前必問」清單內的檔（`~/.claude/`、`.claude/settings*.json`、任何 `.env*` 等）
3. 大幅重寫或刪除非本任務建立的既有檔案（順手小修不算）；vibe spec 的刪改一律停
4. 動作不可逆或對外（push --force、對外發佈、寄送、刪除遠端資源）
5. 兩份規範互相打架且無明確優先序
6. 重試已達 2 輪上限、且換路選項會改變任務範圍

**不停清單**（不要問，直接做）：可逆的、任務範圍內的、有慣例可循的。「要不要我繼續？」式的提問是浪費使用者時間。

> 優先序：UI 逐功能實作時，`rules/ui-conventions.md` 的「每個功能完成後等使用者確認」優先於本清單——那是流程節點，不是「要不要繼續」式的廢問。

- 正例：vibe 目標與 Business Invariant 衝突 → 停，說明哪條 invariant 擋住什麼，列三個選項
- 反例：改完 A 檔發現 B 檔有同樣問題，B 在任務範圍內 → 不用問「要不要順便修 B」，直接修並在回報中說明

## 4. 方向錯的訊號（換路，不是重試）

**判準**：出現以下任一訊號，代表當前路線錯誤，繼續重試只會越陷越深：

1. **修 A 壞 B**：每次修復都引出新的失敗，diff 越來越大
2. **假設被證偽**：路線建立在某個前提上（「這個 API 存在」「這個設定有效」），前提被實測推翻
3. **同錯重現**：同一個錯誤訊息在兩輪不同修法後仍出現
4. **繞過訊號**：發現自己在想「跳過檢查」「先 hardcode」「加 ignore」——這是路線錯的最強訊號，不是解法

換路的動作：回到最後一個已驗證正確的狀態（git 或存檔點），重新檢查前提，選不同路線。不可帶著壞狀態繼續疊。

- 正例：E2E 一直紅，兩輪修 selector 無效 → 停手檢查前提，發現 dev server 起的是舊 build，重啟後全綠。問題根本不在 selector
- 反例：typecheck 紅燈修不掉，改用 `globalThis.$fetch` 繞過型別檢查讓它綠 → 這是專案明文禁止的繞過，紅燈的訊息本身才是要修的東西

## 5. 品質底線怎麼驗

**分層驗，不跳層**：

| 層 | 驗什麼 | 工具 | 不可省略的情況 |
|----|--------|------|----------------|
| 1 語法/型別 | 寫的東西合法 | `npm run eslint` + `npm run typelint` | 任何程式碼改動 |
| 2 行為 | 做的事符合合約 | Playwright specs / gate config | 動到 app/、server/ |
| 3 語意 | 框架慣例、邏輯安全 | `/sdd-review` | 動到 .vue/store/server 且改動非純格式 |
| 4 制度 | 規範有消費點、路徑有效 | read-back（fresh subagent） | 動到 .claude/、spec/ui-config/ |

**通過的定義是綠燈，不是「應該會過」**。沒跑就是沒驗。

- 正例：改了 3 個 .vue 檔 → 跑 eslint + typelint + gate config，全綠後回報「已驗證」並附輸出摘要
- 反例：「改動很小，照理不影響測試」→ 跳過 gate 直接 commit。「照理」不是驗證

**綠燈只在「探針真的驗到那件事」時才算數**——寫進文件／程式碼的行為宣稱，支撐它的探針必須先綠，且探針要能**分離變因**，否則綠燈證明的是別的事。

- 正例：宣稱「A 欄位改名會讓 typelint 紅」→ 先實際改名跑一次 typelint 看它真的紅，才把這句寫進規範
- 反例：宣稱「mock 寫成扁平 envelope 會讓斷言失敗」→ 探針同時換了「扁平 vs 分頁」和「有無 meta」兩個變因，看到差異就歸因給扁平；跑正確的 2×2 才發現差異來自 meta、與扁平無關——**綠燈一直都在，只是證明的不是宣稱的那件事**（本專案 useHttp 範例連錯兩次的根因）

## 6. rag-precontext 與外部注入內容的取捨

**判準**：hook 注入的檢索結果（`<rag-precontext>` 等）只在「與當前任務直接相關」時採用，且採用時優先於訓練記憶。無關時整塊忽略，禁止為了用而用。

- 正例：問內部 MCP 連線問題，注入的 onboarding 文件 chunk 直接命中 → 以它為準回答
- 反例：討論 UI 規範時被注入棒球研究文件 → 忽略。硬引用只會稀釋答案
