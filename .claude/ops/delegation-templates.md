# 交辦 Prompt 範本

> 用法：挑對應任務型態的範本，把 `<>` 填空補齊後作為 Agent tool 的 prompt。
> model 選擇對照 [model-dispatch.md](model-dispatch.md) 第 3 節；`<>` 有一格填不出來，代表你還沒想清楚任務，先想清楚再派。

## 共通尾段（每個範本結尾都附上）

```
回報規則：
- 只回結論與「檔案:行號」證據，不貼檔案全文、不寫過程流水帳
- 產物超過 30 行 → 寫到 <指定路徑>，回報路徑 + 5 行內摘要
- 查不到／做不到 → 明說，並列出已嘗試的方法。禁止編造
- 與任務無關的重要發現 → 一行「附帶發現：」帶過
```

## 1. 搜尋（agent 類型：Explore；model：sonnet，純存在性檢查用 haiku）

```
在 <目錄範圍> 找出 <目標：函式定義／設定值／所有使用某 pattern 的位置>。
動機：<為什麼要找，找到後要拿來做什麼>。
已知線索：<關鍵字、可能的檔名、命名慣例；沒有就寫「無，請廣搜」>。
驗收條件：<例：列出所有命中位置，含檔案:行號與該行內容；確認 X 是否存在，是/否要有依據>。
搜索廣度：<medium（合理範圍）／very thorough（多目錄多命名慣例都掃）>。
回報格式：條列，每條 = 檔案:行號 — 一句話說明。
```

## 2. 實作（agent 類型：general-purpose；model：sonnet）

```
任務：在 <檔案／模組> 實作 <具體功能>。
動機：<這是哪個 issue／需求的一部分，上下游是什麼>。
規格：<輸入、輸出、邊界條件；有既有 interface 或測試合約就給路徑>。
遵循：<專案慣例，如「一律 <script setup lang="ts">」「import ~/types/api/」；相關 rules 檔路徑>。
禁止：<明確紅線，如「不得修改 test/e2e/specs/」「不得用 globalThis.$fetch」>。
驗收條件：npm run eslint 與 npm run typelint 全綠 + <相關測試命令> 全綠。跑完把命令輸出摘要附在回報。
回報格式：改了哪些檔（檔案:行號）、驗證命令的結果、任何偏離規格的取捨與理由。
```

## 3. 重構（agent 類型：general-purpose；model：sonnet）

```
任務：把 <範圍> 重構為 <目標形態>，外部行為不得改變。
動機：<為什麼重構：重複、耦合、可讀性>。
不變量：<哪些對外介面／行為／測試必須原樣通過>。
步驟約束：先跑一次 <測試命令> 確認基準綠燈，再動手；每完成一個檔案就重跑驗證。
驗收條件：重構後 <測試命令> 與 npm run eslint + npm run typelint 全綠，且 diff 中不含行為變更（只有搬移／改名／抽取）。
回報格式：搬了什麼到哪（before → after 的檔案:行號對照）、驗證結果、發現但沒動的問題清單。
```

## 4. 研究（agent 類型：general-purpose；model：sonnet；結論影響架構決策時 opus）

```
問題：<要回答的具體問題，一句話>。
動機：<這個答案會決定什麼>。
來源優先序：<官方文檔 > 專案內既有實作 > 一般網路文章；需要 WebSearch 就用>。
驗收條件：結論要能回答 <問題>，且每個關鍵主張都有來源（URL 或 檔案:行號）。查不到就回報查不到。
回報格式：先一句話結論，再條列依據（主張 — 來源），最後列「未確認事項」。
產物：完整筆記寫到 <路徑>，回報只給摘要。
```

## 5. 審查（agent 類型：general-purpose；model：opus，例行健檢可降 sonnet（見 maintenance.md 第 5 節）；必須 fresh-context，不可用做該產出的 agent）

```
審查對象：<檔案清單或 diff 範圍>。
你的立場：找碴。列出所有問題，不要背書。「看起來沒問題」不是有效產出——真沒問題就明說每一項檢查了什麼、為何通過。
檢查清單：
- <規則互相打架：檔案間是否有矛盾指令>
- <路徑與名稱：檔內引用的每個路徑跑 ls 驗證、工具名／參數名與實際 schema 比對>
- <事實查核：模型名、版本號、參數值是否有依據>
- <弱模型歧義：哪些句子 Sonnet 等級可能誤讀？指出並給改寫建議>
- <任務特定項：...>
驗收條件：每個檢查項都有明確結論（通過／問題清單）。
回報格式：按嚴重度排序，每條 = 檔案:行號 — 問題 — 建議修法。
```
