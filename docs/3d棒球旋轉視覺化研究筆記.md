# 3D 棒球旋轉視覺化：從 result.json 到 three.js 重現

> 目標：拿演算法後端輸出的旋轉分析數據（`result.json`），在前端用 three.js 重現該球的真實旋轉——縫線初始樣貌、轉軸方向、轉向、轉速都要與後端渲染的 `result.gif` 一致。
> 產出：可整包複製到任何專案的解耦模組（`app/components/baseball-spin/`）。

---

## 一、問題定義

後端（bsa-module）分析每一顆球的高速影像後，會輸出該球的旋轉參數。前端要做的是「**把數學重新變回畫面**」：

- 不是播影片、也不是播後端的 gif——是拿參數即時 3D 渲染，之後才能加視角切換（投手/捕手/攝影機）、變速、互動
- 難點不在 three.js 本身，在**座標慣例對齊**：差一個轉置、一個左右乘、一個歸一化，畫面就會「看起來有在轉，但縫線全錯」

## 二、資料合約：result.json 給了什麼

```json
{
  "rpm": 1297.66,
  "axis": [-0.5197, -0.709459, 0.476004],
  "animation": {
    "R_ref": [[0.405, -0.342, 0.848], [-0.914, -0.141, 0.380], [-0.011, -0.929, -0.370]],
    "omega_rad_per_frame": 0.27178053,
    "fps": 500.0
  },
  "spin_dir": { "hhmm": "04:47", "degrees": 143.78 },
  "spin_tilt": { "hhmm": "01:47", "degrees": 233.78 }
}
```

| 欄位 | 意義 | 用途 |
|------|------|------|
| `axis` | **相機座標系**的自轉軸單位向量 | 決定繞哪條軸轉、3D 箭頭指向 |
| `animation.R_ref` | body→camera 的 3×3 旋轉矩陣（row-major），t=0 時球的姿態 | 鎖住縫線初始相位——**缺了它只靠 axis 無法還原真實縫線** |
| `animation.omega_rad_per_frame` | 每一原始影格的旋轉弧度，**帶正負號**（符號＝轉向） | 轉速與方向 |
| `animation.fps` | 原始擷取影格率 | 換算真實角速度 |
| `spin_dir` / `spin_tilt` | 軸方向/傾角的時鐘制讀數 | 數據顯示用（非渲染輸入） |

### 座標慣例（一切正確性的前提）

- 相機系＝右手系 OpenGL 慣例：相機位於 `[0, 0, +3]` 往 −Z 看、+Y 向上、+X 向右、正交投影
- **這與 three.js 的預設慣例完全相同**——所以 `R_ref` 與 `axis` 可以直接用，零轉換
- `R_ref` 只存 6 位小數，捨入後不嚴格正交 → 套用前要正交化（轉 quaternion 再 normalize）

### 資料自帶的交叉驗證

`omega_rad_per_frame × fps × 60 / 2π ≈ rpm` 欄位（三個 sample 都成立，誤差 < 0.1%）。
這條不變量直接寫成單元測試——資料哪天格式改了會第一時間炸出來。

## 三、核心原理：三行數學

任一時刻的球姿態：

```
R(θ) = Rodrigues(axis · sign(omega) · θ) · R_ref
```

1. **`R_ref` 先擺好起始姿態**（由右往左讀），再**繞相機系的 axis 轉 θ**
2. **必須左乘**：axis 是相機系向量，左乘＝「在相機座標裡繞這條軸轉」；寫成右乘會變成繞球體自身座標的軸轉，軸會被 R_ref 帶著跑，全錯
3. θ 從 0 掃到 2π 循環播放＝轉一圈；真實角速度 = `|omega_rad_per_frame| × fps`（rad/s）

一個重要特性：**動畫幾何只由 axis + 符號 + R_ref 決定，轉速在幾何上被約掉**——rpm 只影響播多快，不影響每一格長什麼樣。所以「變速播放」是安全操作，不會破壞正確性。

## 四、技術選型

| 選擇 | 理由 |
|------|------|
| three.js（原生，不用 TresJS） | TresJS 綁 Vue；核心用純 TS + three，換框架只需重寫薄殼 |
| quaternion 而非手刻 Rodrigues 矩陣 | `setFromAxisAngle` 就是 Rodrigues 的 quaternion 形式，正交化/不飄移免費 |
| zod 做 runtime 驗證 | 後端資料防呆＋一次解決 TS strict `noUncheckedIndexedAccess`（parse 後 axis 是 tuple） |
| 3D 箭頭放場景內，不用 SVG 疊圖 | 遮蔽、光影、透視縮短全由渲染管線天然處理（見「踩坑」第 1 條） |

### 模組結構（解耦：整個資料夾複製即用）

```
app/components/baseball-spin/
├── BaseballSpinViewer.vue   # 唯一對外元件（薄 Vue 包裝）
├── README.md                # 搬移規則與 API
└── core/                    # 純 TS + three，零 Vue/Nuxt 依賴
    ├── types.ts             # zod schema：wire(snake_case) → 模組型別(camelCase)
    ├── spin-math.ts         # R_ref→quaternion、R(θ)、rpm 換算
    ├── normalize-model.ts   # glb 中心化＋單位球歸一化
    ├── views.ts             # 視角預設（三視角擴充點，現只有 camera）
    └── scene.ts             # BaseballSpinScene：場景/相機/燈光/動畫迴圈/3D箭頭/dispose
```

關鍵約束：模組內**只用相對 import**。Nuxt 的 `app/components/` 只 auto-import `.vue`，`core/*.ts` 不會被掃——這正好保證複製走不斷鏈；反之若把 core 函式丟進 `app/utils/`（會被全域 auto-import）就破壞可攜性。

## 五、實作詳解

### 5.1 安裝與載入 three.js

```bash
npm i three
npm i -D @types/three
```

```ts
// 核心類別從 'three' 主入口（ESM，可 tree-shaking）
import { Matrix4, OrthographicCamera, Quaternion, Vector3, WebGLRenderer } from 'three'

// 附加模組從 'three/addons/*'——這是套件 exports 定義的官方別名，路徑必須帶 .js
import { GLTFLoader } from 'three/addons/loaders/GLTFLoader.js'
```

不需要改任何 nuxt.config。SSR 注意事項：three 在 server 端 import 不會炸，但沒必要進 server bundle，所以 Vue 元件在 `onMounted` 內動態載入：

```ts
onMounted(async () => {
  const { BaseballSpinScene } = await import('./core/scene') // client 才載＋code-splitting
  scene = new BaseballSpinScene(containerRef.value, { view: props.view, speed: props.speed })
})
```

### 5.2 建場景

three 的心智模型＝ Scene（場景圖）＋ Camera（怎麼看）＋ Renderer（畫到 canvas）：

```ts
this.renderer = new WebGLRenderer({ antialias: true, alpha: true }) // 透明底，對齊 gif
this.renderer.toneMapping = NoToneMapping // 減少與後端 pyrender 的明暗變數
container.appendChild(this.renderer.domElement)

this.camera = new OrthographicCamera()
this.camera.position.set(0, 0, 3) // 後端慣例：相機在 +Z 距離 3
this.camera.lookAt(0, 0, 0)
```

正交相機取景框依容器尺寸重算（`resize()`）：

```ts
const e = 1.5 // halfExtent，對應後端 OrthographicCamera 的 xmag=1.5
camera.left = -e * aspect
camera.right = e * aspect
camera.top = e
camera.bottom = -e
```

球半徑 1、半幅 1.5 → 球直徑固定佔畫面 2/3，與 gif 構圖一致。

場景圖三層嵌套，**層次順序是正確性的一部分**：

```
scene
├── spinGroup          ← 每幀改 quaternion（旋轉套在這層）
│   └── normalizeGroup ← 中心化＋縮放（glb → 原點上的單位球）
│       └── glb 模型
├── axisArrow          ← 3D 轉軸箭頭（不在 spinGroup！球轉、軸不轉）
└── AmbientLight + DirectionalLight
```

旋轉在外、歸一化在內——反過來球就不是繞球心自轉，而是繞偏掉的原點公轉。

### 5.3 載入 glb 與單位球歸一化

```ts
const gltf = await new GLTFLoader().loadAsync(url, e => onProgress?.(e.loaded, e.total))
```

後端渲染前會把模型歸一化成單位球（trimesh 的 `load_unit_sphere_trimesh`），`R_ref` 是基於那個狀態算的，前端必須做一模一樣的事：

```ts
// 兩遍統計：(1) 所有頂點平均位置 = center  (2) 中心化後的平均頂點半徑 = meanRadius
// 然後套在 wrapper Group（不改頂點資料），等價 Python 的 (vertices - center) / radius：
wrapper.scale.setScalar(1 / meanRadius)
wrapper.position.copy(center).multiplyScalar(-1 / meanRadius)
```

少這步的症狀：球轉起來偏擺（不繞球心）＋畫面大小對不上。

### 5.4 讓球照數據轉

```ts
// R_ref（3×3 row-major）→ quaternion
// 陷阱：Matrix4 內部儲存是 column-major，不可直接塞 .elements！
// 要走 m.set(...)——它收 row-major 引數。normalize() 吸收 6 位小數的非正交誤差。
const m = new Matrix4().set(
  r00,
  r01,
  r02,
  0,
  r10,
  r11,
  r12,
  0,
  r20,
  r21,
  r22,
  0,
  0,
  0,
  0,
  1,
)
const qRef = new Quaternion().setFromRotationMatrix(m).normalize()

// 每幀姿態：R(θ) = Rodrigues(axis·θ)·R_ref 的 quaternion 版
qSpin.setFromAxisAngle(axisSigned, theta)
out.multiplyQuaternions(qSpin, qRef) // qSpin 在前＝左乘；寫反＝繞 body 軸，錯
spinGroup.quaternion.copy(out)
```

動畫迴圈用 rAF，但**角度用時間累加、不用幀數**——60Hz/120Hz 螢幕、分頁節流都不影響轉速：

```ts
const delta = (now - lastTime) / 1000
theta = (theta + delta * radPerSec * speed) % (2 * Math.PI)
// radPerSec = |omega_rad_per_frame| × fps（真實角速度）；speed = 播放倍率
```

### 5.5 3D 轉軸箭頭

不畫 2D 箭頭模擬 3D，而是把箭頭當**真 3D 物件**放進場景：

```ts
function buildAxisArrow(): Group {
  const material = new MeshStandardMaterial({ color: 0xFACC15, roughness: 0.55 })
  const headLen = 0.4
  const shaft = new Mesh(new CylinderGeometry(0.035, 0.035, 4 - headLen), material)
  shaft.position.y = -headLen / 2
  const head = new Mesh(new ConeGeometry(0.12, headLen), material)
  head.position.y = 2 - headLen / 2 // 錐尖端剛好在 +2（球心往外 2 個球半徑）
  // ...
}
```

- 尺寸抄後端 `set_rotation_axis(length=4.0, radius=0.035)`：全長 4 個球半徑、單頭
- 幾何沿 +Y 建（three 的 Cylinder/Cone 預設軸向），指向轉軸只要一行：
  `arrow.quaternion.setFromUnitVectors(UP, axisSigned)`（+Y → 帶號軸的最短旋轉）
- `axisSigned = normalize(axis) × sign(omega)`：轉速為負就反向，箭頭永遠指右手定則正端
- 掛在 scene 根、不掛 spinGroup：**自轉軸是不變量**，球轉、軸不動
- 穿球遮蔽（中段藏進球裡）、圓柱明暗、軸朝鏡頭時的壓縮感——全部是 depth buffer＋光照＋正交投影的天然結果，零手寫數學

### 5.6 Vue 薄包裝

核心 class 不認識 Vue，元件只做四件事：

```ts
// (1) onMounted：dynamic import 建場景＋載模型
// (2) props → class 方法
watch(() => props.data, d => d && scene?.setData(d))
watch(() => props.speed, s => scene?.setSpeed(s))
watch(() => props.showAxisArrow, v => scene?.setAxisArrowVisible(v))
// (3) ResizeObserver → scene.resize()
// (4) onUnmounted → scene.dispose()
//     不做會洩漏 WebGL context（瀏覽器上限約 8–16 個，超過開始丟棄）：
//     cancelRAF → traverse dispose geometry/material/texture
//     → renderer.dispose() → forceContextLoss() → 移除 canvas
```

換框架（React/Svelte）只需重寫這層薄殼，5.1–5.5 原封不動。

## 六、驗證方法與結果

分三層，由內而外：

**1. 單元測試（數學層，vitest）**——渲染之前先確保數學對：

| 測試 | 抓什麼錯 |
|------|---------|
| `omega × fps` 換算 ≈ rpm 欄位（三 sample） | 資料解讀錯誤 |
| R_ref → quaternion → 矩陣 roundtrip | Matrix4 row/column-major 塞反 |
| `R(0) ≡ R_ref`、`R(2π) ≡ ±R_ref` | 公式基本錯誤 |
| **轉軸不變量**：`R(θ)·(qRef⁻¹·axis) ≈ axis` 對任意 θ | **左乘寫成右乘**（最致命的錯，畫面照樣會動） |
| omega 正負 → 互為逆旋轉 | 轉向符號 |

**2. 視覺比對（demo 頁 `/spin-demo`）**——與後端 `result.gif` 並排：

- t=0 凍結姿態 vs gif 第一幀 → 驗 R_ref（縫線初始樣貌）✅ 三 sample 全吻合
- 連續截圖 vs gif 10%/20% 進度幀 → 驗轉向（同向不鏡像）✅
- 球佔畫面比例 → 驗歸一化＋halfExtent ✅
- 3D 箭頭方向/長度/遮蔽 vs gif 黃箭頭 ✅

**3. 生命週期驗證**——元件反覆掛載/卸載無 WebGL context 警告。

### 驗證中的發現

- **gif 的 slow_factor 是 60，不是文件預設的 8**：由幀數反推（83/69/86 幀 ÷ 30fps ≈ 真實一圈時間 × 60），三個 sample 都吻合。demo 的「對照 gif」速度因此用 1/60x。
- **spin_tilt 與軸投影的換算關係**：螢幕投影角 ≡ `90° − spin_tilt.degrees`（mod 360，三 sample 實證）。gif 的黃箭頭是 3D 軸的正交投影；jsjh-2026-frontend 那種時鐘面板直接 `rotate(spin_tilt.degrees)` 是另一種呈現（數據視覺化），兩者不能混用。

## 七、踩坑記錄（迭代歷程）

1. **指針走了三個版本**：
   - v1 SVG 疊圖＋`rotate(spin_tilt.degrees)` → 角度與 gif 差約 18°（tilt 讀數 ≠ 投影角）
   - v2 SVG＋軸投影角＋手推遮蔽幾何（朝鏡頭端從球面浮出點 ρ 畫起、背側只畫剪影外）→ 正確但一堆手寫數學
   - v3 **真 3D 物件進場景** → 所有手寫幾何刪光，效果反而完全正確。教訓：**要模擬 3D 效果時，先想想能不能直接用 3D**。
2. **`Matrix4.set` 收 row-major 引數、內部存 column-major**：直接塞 `.elements` 會轉置出錯，用 roundtrip 測試把關。
3. **quaternion 乘序**：`multiplyQuaternions(qSpin, qRef)` 才是左乘。寫反了球「看起來也在轉」——只有轉軸不變量測試抓得到。
4. **dev SSR 抓不到 public/ 靜態檔**：`useFetch('/samples/...json')` 在 SSR 走 nitro 內部管線 → 404（client 端正常，所以症狀是「頻繁但不是每次」）。demo 頁 fetch 靜態檔加 `server: false` 解決。
5. **headless 驗證要開軟體渲染**：Playwright headless Chromium 建不出 WebGL context，launch 加 `--enable-unsafe-swiftshader`。
6. **glb 17.8MB 首載慢**：GLTFLoader `onProgress` → loading slot 顯示進度；HTTP cache 二次即快；draco/meshopt 壓縮列為未來優化。

## 八、未來擴充：三視角

`core/views.ts` 已留擴充點。原理：R_ref 是 body→camera，姿態管線輸出的是相機系 quaternion，**換視角＝對最終姿態再左乘一個視角旋轉**（`qFinal = qView · qCam`）。後端有相機校正（.pkl）時 result.json 會多出 `axis_world` 與 `views.{camera,pitcher,batter}`，屆時在 `VIEW_PRESETS` 補 preset 即可，`BaseballSpinViewer` 的 `view` prop 已就緒。
